     Fix: Savings Rate Shows 0.0% When Spending Exceeds Income

     Context

     The "Saving rate" bar in the Spending Personality card (Analytics & Reports
      screen) always shows 0.0% even when the user has recorded income
     transactions. This is misleading because 0.0% could mean either "no income
     recorded" or "spending exceeds income" — but the two situations look
     identical.

     Root cause: The formula in _buildSpendingPersonality() clamps the saving
     rate to [0.0, 1.0]:
     final savingRate = income > 0 ? ((income - expense) / income).clamp(0.0,
     1.0) : 0.0;
     When expense >= income (even by ₹1), the raw rate is ≤ 0, and .clamp(0.0, 
     1.0) forces it to 0.0, hiding the fact that the user is overspending.

     Evidence from code tracing:
     - The saving rate section is only rendered when income > 0 (line 2078: if 
     (income > 0) ...), so the fact it IS showing means the user has income
     - saveTransaction() in transaction_controller.dart stores send amounts as
     negative: finalAmount = isSend ? -amount : amount; totalIncome /
     totalExpense in analytics.dart handle this correctly with .abs() for
     expenses
     - The screenshot shows 0.0% with the Frequent Spender personality and top
     merchants like Loan Payment ₹10,243 + Hostel Rent ₹10,000 + EMI Payment
     ₹4,691 — total expenses clearly exceed any typical monthly income

     ---
     Fix

     File: lib/Screens/analytics.dart, method _buildSpendingPersonality()
     (~lines 2021–2103)

     1. Compute the raw (unclamped) rate

     final rawRate = income > 0 ? (income - expense) / income : 0.0;
     final isDeficit = rawRate < 0;
     final savingRate = rawRate.clamp(0.0, 1.0); // still used for the progress 
     bar value

     2. Choose label color — red when overspending, personality color otherwise

     final rateColor = isDeficit ? Colors.redAccent : p.color;

     3. Show the actual percentage (not the clamped 0.0%)

     Replace the current percentage Text widget:
     // OLD
     Text(
       '${(savingRate * 100).toStringAsFixed(1)}%',
       style: TextStyle(fontSize: 12.sp, fontWeight: FontWeight.w700, color:
     p.color),
     ),

     // NEW
     Text(
       '${(rawRate * 100).toStringAsFixed(1)}%',
       style: TextStyle(fontSize: 12.sp, fontWeight: FontWeight.w700, color:
     rateColor),
     ),

     4. Tint the progress bar red on deficit

     // OLD
     valueColor: AlwaysStoppedAnimation(p.color),

     // NEW
     valueColor: AlwaysStoppedAnimation(rateColor),

     5. Add a deficit hint text below the bar when overspending

     if (isDeficit) ...[
       SizedBox(height: 4.h),
       Text(
         'Spending exceeds income by ${((-rawRate) * 100).toStringAsFixed(1)}%',

       ),

● The fix works differently now:

  - Before: Found the single largest credit, then checked keywords on it after
  the fact → if EMI was the only large credit, filtering it left nothing and the
   widget disappeared
  - After: Strips EMI/loan credits out of the candidate pool first, then
  recomputes the median and max on the remaining credits → if a real salary
  exists, the statistical check runs on clean data and detects it correctly

  The keyword check is also tighter now — only recipientName is checked (not
  note or category, which could be "Transfer" or "NEFT" for a perfectly
  legitimate salary). The pattern also drops transfer/neft/imps/rtgs/refund
  which are payment method words that a user could have in a real salary record.

  1. Transaction amount sign convention — the fact that send amounts are stored
  negative is easy to miss and will cause wrong totals in any new analytics code
   that doesn't call .abs() correctly.
  2. Salary detection pattern — the two specific pitfalls:
  filter-before-statistics (not after) to avoid the widget going blank, and only
   match keywords against recipientName (not category/note) to avoid false
  negatives on legitimate NEFT salary transfers.
